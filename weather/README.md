# 비상근무 알림 시스템 기획서

## 1. 프로젝트 개요

날씨 특보를 실시간에 가깝게 감지하고, 특보 유형과 지역, 부서별 기준에 따라 비상근무 대상자에게 자동으로 알림을 전송하는 시스템을 만든다.

이 프로젝트의 목표는 아래 3가지다.

- 특보 정보를 빠르게 수집하고 최신 상태로 유지한다.
- 부서별로 필요한 특보만 선택해서 알림 정책을 다르게 운영한다.
- 순번 기반 비상근무자가 사전에 알림을 받아 대응할 수 있게 한다.

## 2. 기술 스택

- FE: HTML, CSS, JavaScript
- BE: Python, FastAPI
- DB: SQLite

## 3. 핵심 사용자

- 관리자
- 부서 담당자
- 비상근무 대상 직원

## 4. 핵심 기능

### 4.1 날씨 특보 수집

- 날씨 API를 사용해 강풍주의보, 호우주의보, 호우경보 등 특보 정보를 수집한다.
- 5분마다 자동 갱신한다.
- 최신 수집 결과를 기준으로 화면과 알림 상태를 갱신한다.

### 4.2 지역 선택

- 사용자가 원하는 지역을 선택할 수 있어야 한다.
- 지역별 특보를 구분해서 보여준다.
- 지역 선택값은 부서별 정책과 연동할 수 있어야 한다.

### 4.3 비상근무 명단 등록

- 직원 명단을 파일 업로드 또는 수기 입력으로 등록할 수 있어야 한다.
- 지원 파일 형식은 다음을 기준으로 한다.
  - `xlsx`
  - `csv`
  - `hwp`
  - `hwpx`
  - `pdf`
- 업로드 후에는 컬럼 매핑 또는 미리보기 확인이 가능해야 한다.
- 수기 입력은 단건 등록과 다건 입력을 모두 고려한다.

### 4.4 부서별 특보 기준 설정

- 부서마다 알림을 받을 특보 유형을 선택할 수 있어야 한다.
- 예를 들어 어떤 부서는 호우경보만 받도록 하고, 어떤 부서는 강풍주의보도 받도록 설정할 수 있어야 한다.
- 설정은 저장되며 나중에 수정할 수 있어야 한다.
- 수정 버튼을 통해 기존 정책을 재설정할 수 있어야 한다.

### 4.5 순번 기반 알림

- 비상근무 순번이 지나가면 다음 순번에게 사전 알림이 가야 한다.
- 알림 수단은 문자 또는 카카오톡을 우선 고려한다.
- 순번 교체 시점과 사전 알림 시점을 분리해서 관리한다.

### 4.6 즉시 알림

- 선택한 날짜의 특보가 실제로 발생하면 해당 순번에게 즉시 알림을 보낸다.
- 지역과 특보 유형이 정책 조건에 맞아야 알림을 발송한다.
- 중복 발송을 막기 위한 발송 이력 관리가 필요하다.

## 7. 처리 흐름

### 7.1 특보 감지 흐름

1. 5분마다 날씨 API를 호출한다.
2. 선택 지역의 특보를 조회한다.
3. 저장된 부서 정책과 비교한다.
4. 조건에 맞으면 알림 대상자를 찾는다.
5. 사전 알림 또는 즉시 알림을 발송한다.
6. 발송 결과를 로그로 남긴다.

### 7.2 명단 등록 흐름

1. 관리자가 파일 또는 수기로 명단을 입력한다.
2. 시스템이 형식을 확인한다.
3. 필요한 경우 컬럼 매핑을 보여준다.
4. 저장 후 순번과 부서 정책에 연결한다.

## 15. 실행 안내

- FE는 브라우저에서 `frontend/index.html`을 열어 확인한다.
- BE는 FastAPI 환경에서 `uvicorn app.main:app --reload`로 실행한다.
- DB는 SQLite 파일 `backend/data/weather.db`를 사용한다.

## 16. 아주 쉽게 실행하기

### 1) 백엔드 실행

PowerShell을 열고 아래를 그대로 실행한다.

```powershell
cd C:\Users\admin\Desktop\DAY3\weather\backend
python -m uvicorn app.main:app --reload
```

### 2) 브라우저로 프런트 열기

탐색기에서 아래 파일을 더블클릭한다.

- `C:\Users\admin\Desktop\DAY3\weather\frontend\index.html`

또는 간단한 로컬 서버를 쓰고 싶으면 아래를 실행한다.

```powershell
cd C:\Users\admin\Desktop\DAY3\weather\frontend
python -m http.server 5500
```

그 다음 브라우저에서 아래 주소를 연다.

- `http://127.0.0.1:5500`

### 3) 정상 동작 확인

- 백엔드 상태 확인: `http://127.0.0.1:8000/api/health`
- 프런트 화면에서 지역, 명단, 정책, 업로드 목록이 보이면 정상
- 저장 버튼을 눌러서 목록이 갱신되면 정상

### 4) 접속이 안 될 때 확인할 것

- `127.0.0.1`은 이 PC에서만 열린다.
- 다른 PC나 휴대폰에서 보려면 Cloudflare Tunnel 같은 공개 URL을 써야 한다.
- GitHub Pages에서 프런트를 쓸 거면, 화면 상단의 `백엔드 URL` 칸에 공개된 Cloudflare 주소를 넣어야 한다.
- 로컬에서만 테스트할 때는 기본값 `http://127.0.0.1:8000`을 그대로 쓰면 된다.

### 5) 종료

- 실행한 터미널에서 `Ctrl + C`를 누르면 끝난다.
